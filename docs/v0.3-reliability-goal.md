# v0.3 Streaming Reliability 1.0

MacStream v0.3 proves the existing single-destination streaming path before
adding camera effects, presenter segmentation, or multi-destination output.
Passing unit tests is necessary, but the release also requires evidence from
the signed application against a real RTMP ingest endpoint.

## In scope

- Runtime RTMP interruption detection and bounded recovery.
- Record-while-streaming ownership and shutdown behavior.
- A/V timing and delivery-health observability.
- Bounded media queues under encoder or network pressure.
- Long-session CPU, memory, thermal, FPS, and dropped-frame evidence.
- A signed and notarized DMG for first-time installation.
- Product-focused README and release documentation.

## Deferred

- Multi-destination streaming.
- Presenter Cutout and automatic person segmentation.
- Center Stage, Portrait, Studio Light, and Background Replacement controls.
- Editing, trimming, and post-recording workflows.
- AI-assisted setup, transcription, and summaries.

## Automated gates

1. Default and HaishinKit test suites pass from the repository root.
2. A live RTMP failure enters recovery without accepting destination edits or
   duplicate starts.
3. Recovery uses bounded retry and closes each failed publisher before
   replacing it.
4. Cancelling recovery leaves the stream offline and ignores late completion.
5. A stream-owned recording remains active through successful RTMP recovery.
6. Exhausted recovery stops a stream-owned recording exactly once; a manually
   started recording remains independently owned.
7. Video and audio append queues remain bounded. Pressure drops samples and
   increments health counters instead of accumulating unbounded work.
8. Stream keys remain redacted in events, reports, logs, and exported evidence.
9. The actual HaishinKit publisher sends paced video to a localhost RTMP ingest;
   the decoded output is H.264 at the configured dimensions with at least 90%
   of expected frames.

## Packaged-app scenarios

Run these with the signed release candidate and a real ingest endpoint:

| Scenario | Duration | Required result |
| --- | ---: | --- |
| Screen + Webcam RTMP | 60 minutes | Remote composition remains valid; no frozen or black output. |
| Screen + Webcam RTMP plus recording | 60 minutes | Stream and `.mov` remain playable with stable A/V timing. |
| Screen-only RTMP plus recording | 60 minutes | No camera session starts; stream and recording remain valid. |
| Short network interruption | 5 seconds | Stream recovers within the configured retry budget. |
| Longer network interruption | 30 seconds | Stream either recovers or reaches a truthful failed state without hanging. |
| Repeated lifecycle | 20 start/stop cycles | No stuck state, duplicate writer, or orphan publisher. |

For recorded and remotely captured media, measure sync at 10, 30, and 60
minutes. Target absolute A/V drift at or below 100 ms; treat drift above 150 ms
as a release blocker.

## Performance evidence

Use the same Mac, source devices, output profile, and ingest endpoint for every
comparison. Record:

- Capture FPS versus target FPS.
- Dropped video frames and rejected RTMP audio appends.
- Outbound bitrate and bytes per second.
- Process CPU, resident memory, thread count, thermal pressure, and Low Power
  Mode state.
- Reconnect start time, result time, attempt count, and total interruption.

Nominal long sessions must not show sustained zero capture FPS, sustained audio
delivery stalls, unbounded memory growth, or queue growth beyond configured
limits. Compare CPU against `docs/performance-baseline.md` on the same machine;
do not use hardware-specific CPU percentages as CI thresholds.

Store manual results under `benchmarks/` using dated CSV and Markdown files.
Never include stream keys or unredacted destination URLs.

Use `script/record_reliability_metrics.sh` to capture repeatable process CPU,
resident memory, thread count, and elapsed-time samples while the app's
session report records media and recovery health.

## Distribution gate

The release workflow must publish:

- `MacStream-vX.Y.Z-macos-arm64.dmg` for first-time installation.
- `MacStream-vX.Y.Z-macos-arm64.zip` for Sparkle updates.
- SHA-256 checksum files for both artifacts.

The DMG must contain `MacStream.app` and an `/Applications` link, be Developer
ID signed, notarized, stapled, and pass Gatekeeper assessment. The ZIP must
continue to pass Sparkle EdDSA signing and appcast verification.
