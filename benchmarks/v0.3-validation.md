# v0.3 Validation Ledger

Date: 2026-07-16

This ledger records release-gate evidence for the v0.3 streaming reliability
goal. It contains no destinations, stream keys, credentials, or captured media.

- `MAC_STREAM_ENABLE_HAISHINKIT=1 swift test`: 361 tests passed.
- `swift build -c release --arch arm64`: passed.

## Real Publisher Integration

The actual HaishinKit publisher was exercised against an FFmpeg RTMP listener
on localhost with paced synthetic video:

| Duration | Target | Decoded at ingest | Result |
| ---: | ---: | ---: | --- |
| 5 seconds | 75 frames | 74 H.264 frames | Passed |
| 60 seconds | 900 frames | 899 H.264 frames | Passed |

Both runs produced 640x360 H.264 video, reported nonzero publisher outbound
bytes, accepted at least 90% of submitted frames, and drained the publisher
append queue on close. CI and release automation now run the five-second form
through `script/run_rtmp_integration.sh`.

This closes the real publisher-to-ingest integration gate. It does not exercise
the packaged app, ScreenCaptureKit, camera capture, compositor, microphone or
system audio, recording, remote network conditions, or runtime reconnection.
Those remain release gates below.

## Packaged Capture And Recording QA

The arm64 HaishinKit release package was exercised with the built-in MacBook Pro
Camera, a wireless microphone, and the `Screen + Webcam` scene.

- The idle webcam preview released the camera before the media pipeline started
  its recording capture session.
- Camera frames continued to refresh the composed output when the captured
  screen was static.
- Wireless microphone buffers were normalized to interleaved PCM before being
  appended to the recording writer.
- Recording stopped cleanly and the studio returned to `Ready` without writer
  failure logs.
- Two resulting `.mov` files were validated with FFprobe:

| Duration | Video | Audio | Result |
| ---: | --- | --- | --- |
| 87.47 seconds | H.264, 1920x1080 | AAC stereo, 48 kHz | Passed |
| 118.41 seconds | H.264, 1920x1080 | AAC stereo, 48 kHz | Passed |

Extracted frames showed both the selected screen and the built-in camera in the
recorded composition. This closes the short packaged `Screen + Webcam`
recording blocker. It does not replace the required long-duration RTMP and
record-while-streaming runs.

## Required Before Tagging v0.3.0

- Grant fresh Camera, Microphone, and Screen Recording permissions to the exact
  release candidate and verify recovery after relaunch.
- Run 60 minutes of real Screen + Webcam RTMP publishing against a private ingest.
- Run 60 minutes of real RTMP publishing with record-while-streaming enabled.
- Capture CPU, RSS, thread count, dropped frames, queue depth, recovery metrics,
  and A/V drift for both long sessions.
- Interrupt the network for approximately 5 seconds and 30 seconds; verify
  bounded recovery, truthful state, and recording ownership.
- Run 20 packaged start/stop lifecycle cycles without stale capture, writer, or
  publisher state.
- Re-run the signed release workflow from the final v0.3 commit and verify the
  notarized/stapled DMG and Sparkle ZIP.
- Install the final release artifacts on a clean macOS user or machine and
  repeat the permission and capture smoke checks.

Do not create or push the v0.3.0 tag until these remaining gates are recorded as
passed or the release owner explicitly changes the acceptance criteria.
